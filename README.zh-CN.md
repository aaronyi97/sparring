# Sparring

[English README](README.md)

对练场（Sparring）是一个 local-first 的决策工具。重大决策前，它先让 3 个不同家族的 AI 吵起来，再由 AI 教练拿真实的吵点逼你想清楚，最后带你走出一份决策备忘录。

它不是群聊，不是投票机，也不是替你拍板的工具。重点不是收集三个意见然后取平均，而是把几个独立、够强的推理者**真正分歧的地方**摊开，在任何综合揭示之前先逼你自己表态，并且把最终判断权留在你手里。

## 当前状态

这是 v1，local-first，已作为可运行的本地版本完成（里程碑 M1–M5）。请把它当成一个诚实的 v1，而不是打磨好的成品：它能用真实模型跑通全链路，有 58 个通过的测试，部署脚本写好了但还没在真实主机上跑过。

它的定位是在你自己的机器上、用你自己的模型凭证来跑。没有托管服务。

## 它解决什么

核心循环是七步：

1. **题型门卫** —— 宽进。纯查事实的问题（"X 的首都是什么"）会被退回让你改写；真正的决策才放行。
2. **异构扇出** —— 3 个 AI 家族对同一个问题并行、独立作答，彼此不知道对方（`n_of_m = 2`，先到先用，一个慢模型不会拖住整轮）。
3. **分歧地图** —— 基于这些独立答案，给出共识、分歧点，以及一个整体的决策歧义度评分。
4. **教练** —— 五轨道苏格拉底式教练，拿**真实**分歧带你走自己的思路，背后有一个独立收口判定器，防止自证偏置。
5. **立场卡** —— 在任何揭示之前，强制你先写下自己的立场。这一步不可跳过。
6. **对照揭示** —— 你的推理 vs 各方综合后的观点，逐点 diff。
7. **决策备忘录** —— 一份可导出、可分享的记录：你做了什么决定、为什么。

跨会话还有一个可选的定性画像，观察你的决策模式（opt-in 同意，一键删除）。

## 它不做什么

- 不替你拍板，也不投决定性的一票。
- 不是模型之间的群聊，也不是人气投票。
- 在你表态之前，不让模型看到彼此的答案。
- 不作为托管的多租户服务运行，也不把你的数据同步到任何地方。
- 没有你自己的凭证就不调用模型；端点缺失会 fail closed 直接报错，而不是静默回退。
- 不是放密钥、客户数据，或任何你不想离开本机的东西的地方。

## 快速开始

要求：

- Python 3.11 或更新版本。
- Node.js（前端用）。
- 你自己的模型凭证（八路端点，见 `.env.example`）。

后端：

```bash
python3.11 -m venv .venv && .venv/bin/pip install -r requirements.txt
cp .env.example .env          # 然后填入你自己的端点和 key
.venv/bin/python -m uvicorn sparring.api:app --port 8788
```

前端（另开一个终端）：

```bash
cd web && npm install && npm run dev   # 代理 /api 到 8788
```

种一个邀请码（登录用）：

```bash
.venv/bin/python -m sparring.seed_invite 5 "first batch"
```

测试、CLI 运行，以及真实 HTTP 全链路检查：

```bash
.venv/bin/python -m pytest -q
.venv/bin/python -m sparring.cli "你的真实决策问题"
.venv/bin/python tools/e2e_live.py     # 需要后端在跑
```

## 模型路由

三个默认贡献者刻意做成异构——来自三个不同家族、一家一个——这样分歧才是真的，而不是同一种风格的三个变体。另有一个模型当教练，更轻的模型负责门卫、分歧分析、收口判定器和最终综合。

换模型只改一行：`config.yaml` 里的 `roles` 块。每次真实模型调用都会记进成本台账 `data/sparring.db::cost_ledger`（按 request id 幂等），所以你能看到每次会话实际花了多少。

## 架构

- **后端：** FastAPI + Server-Sent Events 做流式输出，底层是 SQLite（WAL 模式）。
- **前端：** React 19 + Tailwind CSS v4，用 Vite 构建，通过 SSE 和后端通信，带心跳和硬超时。
- **鉴权与限额：** 邀请码登录（客户端不能自报身份）、每用户每日配额，以及 opt-in 的画像观察。
- **单 worker：** 后端在进程内保留了一部分状态（一个后台综合任务注册表），所以 v1 以单 uvicorn worker 运行。横向扩容是 v2 的议题——把会话搬到 Postgres，综合任务下沉到队列。完整链路见 `deploy/`（v1 部署脚本写好了但没跑；真正上线需要你自己来）。
